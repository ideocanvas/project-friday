# Improvement Plan: Friday vs WhatsApp Chatbot Reference

## Executive Summary

The reference WhatsApp chatbot (Lucy) produces better responses than Friday primarily due to:
1. **Response cleaning pipeline** — strips thinking tags, tool artifacts, and truncates
2. **Knowledge extraction** — automatically learns user preferences from conversations
3. **Two-tier response strategy** — simple text vs agentic tool calling with smart routing
4. **Learned knowledge injection** — feeds back user knowledge into LLM context
5. **Higher tool iteration limit** — 15 rounds vs Friday's 5

Below are prioritized improvements to bring Friday's response quality up to par.

---

## Phase 1: Response Cleaning Pipeline (High Impact, Low Effort)

**Problem:** Friday sends raw LLM output to WhatsApp, including `<think...>` blocks, "Based on my search results..." artifacts, and potentially overly long messages.

**Changes:**
- Create `core/response-cleaner.ts` based on the reference's `utils/responseCleaner.ts`
  - `removeThinkingTags()` — strip `<think...</think` blocks
  - `removeToolCallArtifacts()` — strip phrases like "Let me search for that information", "Based on my search results"
  - `normalizeWhitespace()` — collapse 3+ newlines, trim, remove double spaces
  - `shortenForWhatsApp()` — truncate to 1000 chars with ellipsis
  - `cleanLLMResponse()` — composes all the above
- Integrate into `agentLoop()` in `core/message-processor.ts` — clean the final text response before returning
- Also integrate into `task-manager.ts` agent loop for background task results

**Files to modify:**
- `core/message-processor.ts` — apply cleaning in `agentLoop()` return path (line ~808)
- `core/task-manager.ts` — apply cleaning in its agent loop

**Files to create:**
- `core/response-cleaner.ts`

## Phase 2: Knowledge Extraction & Injection (High Impact, Medium Effort)

**Problem:** Friday only knows what's in `profile.json` (name, location, timezone) and raw `memory.log` entries. It has no mechanism to learn user preferences, important dates, or contextual facts from conversations.

**Changes:**

### 2a. Extend UserProfile with `knowledge` field
- Add `knowledge?: Record<string, { value: string; source: string; lastUpdated: string }>` to `UserProfile` interface
- Backward compatible — optional field, defaults to `{}`

### 2b. Create `core/knowledge-extractor.ts`
- Port the reference's `KnowledgeExtractionService`
- `extractKnowledgeFromConversation(phone)` — sends last 10 memory entries to LLM with extraction prompt, returns structured `[{topic, value, confidence, context}]`
- `processAndStoreKnowledge(phone, items)` — filters (confidence >= 0.5), merges into profile.knowledge
- `scanConversationForKnowledge(phone)` — orchestrates extraction + storage
- Trigger condition: every 5th message in a conversation, or when memory exceeds 20 entries

### 2c. Inject learned knowledge into system prompt
- In `buildSystemPrompt()`, read `profile.knowledge` and format it into the prompt:
  ```
  ## Known User Information
  - Prefers black coffee (learned from conversation)
  - Birthday is in March (learned from conversation)
  - Works in finance (learned from conversation)
  ```
- This gives the LLM context about the user without needing to search memory

### 2d. Add `<learn>` tag protocol for inline knowledge capture
- In `agentLoop()`, after receiving the final text response, scan for `<learn topic="..." source="...">...</learn>` tags
- Parse, store in profile.knowledge, strip tags before sending to user
- Add instruction to `soul.md` telling the LLM it can use `<learn>` tags to remember things

**Files to modify:**
- `core/message-processor.ts` — extend UserProfile, modify buildSystemPrompt(), add learn tag parsing in agentLoop()
- `soul.md` — add knowledge capture instructions

**Files to create:**
- `core/knowledge-extractor.ts`

## Phase 3: Smart Tool Routing / Two-Tier Response (Medium Impact, Medium Effort)

**Problem:** Friday always runs the full agent loop with all tools for every message, even simple greetings or confirmations. This is slow and can cause unnecessary tool calls.

**Changes:**

### 3a. Add simple text response path
- For messages that don't need tools (greetings, confirmations, simple questions), skip the tool-calling agent loop entirely
- Add `shouldUseTools(message, history)` heuristic similar to reference's `shouldUseSearch()`:
  - Check for search-related keywords (current, latest, news, weather, how to, search, find, look up, etc.)
  - Check if user is asking about real-time data or needing external information
  - If no tool keywords detected AND message is short (< 50 chars), use simple text generation
- Simple text path: send system prompt + history + user message to LLM with **no tools**, lower temperature

### 3b. Increase tool iteration limit
- Current: `AGENT_MAX_ITERATIONS = 5` — too low for multi-step research tasks
- Increase to `10` (configurable via env var, already is, just change default)
- The reference uses 15; 10 is a good middle ground for local models

### 3c. Triage refinement
- The existing triage system classifies into rapid_response / background_task / skill_generation
- Within `rapid_response`, add the simple-vs-tool-calling decision
- Flow: triage → if rapid_response → shouldUseTools? → simple text OR agent loop with tools

**Files to modify:**
- `core/message-processor.ts` — add `shouldUseTools()`, add simple text path in `processMessage()`
- `.env.example` — update `AGENT_MAX_ITERATIONS` default to 10

## Phase 4: Memory System Improvements (Medium Impact, Higher Effort)

**Problem:** Memory search is simple keyword matching. Memory grows unbounded. No summarization.

**Changes:**

### 4a. Memory pruning
- Add max memory entries per user (e.g., 500 entries)
- When exceeded, prune oldest entries while keeping the most recent 400
- Add periodic cleanup (can be added to janitor.ts)

### 4b. Memory summarization (stretch goal)
- The constant `CONTEXT_SUMMARY_THRESHOLD_BLOCKS` is defined but not used
- When context blocks exceed the threshold, use LLM to summarize older blocks into a single message
- Store summary in profile or as a special memory entry

**Files to modify:**
- `core/message-processor.ts` — add pruning logic to memory append, implement summarization
- `core/janitor.ts` — add memory cleanup task

---

## Phase 5: Error Handling & Reliability (Lower Priority)

**Problem:** Less robust than reference's multi-layer fallback chains.

**Changes:**
- Add `isConfigured()` pattern — check if LLM service is available before attempting calls, with graceful fallback
- Add per-tool try/catch in agent loop (reference isolates each tool execution)
- Fix the `ALILED_NUMBERS` typo in gateway.ts (line 34)

**Files to modify:**
- `core/message-processor.ts` — isolate tool execution errors
- `core/llm-client.ts` — add health check / isConfigured()
- `core/gateway.ts` — fix typo

---

## Implementation Order

1. **Phase 1** (Response Cleaning) — immediate visual improvement, ~30 min
2. **Phase 2** (Knowledge Extraction) — biggest quality improvement, ~2-3 hrs
3. **Phase 3** (Smart Tool Routing) — speed improvement for simple queries, ~1 hr
4. **Phase 4** (Memory Improvements) — reliability, ~1 hr
5. **Phase 5** (Error Handling) — robustness, ~30 min

## Notes

- Phases 1-3 address the core "better response" issue
- Phase 1 alone will likely fix the most visible quality issues (thinking tags leaking, tool artifacts)
- Phase 2 is the most impactful for making Friday feel "smarter" — it will remember user context across conversations
- All changes are backward compatible and don't require infrastructure changes
